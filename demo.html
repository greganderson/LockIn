Widget REST API v2 — Complete Reference

The Widget REST API provides programmatic access to all resources within the Widget platform, including widgets, widget collections, widget templates, and the associated metadata records that describe the provenance, lifecycle state, and ownership of each of those resources, and this document constitutes the authoritative reference for version 2 of the interface, which supersedes version 1 in all respects except where explicitly noted in the migration appendix. All requests must be made over HTTPS to the base URL https://api.example.com/v2, and any request made over plain HTTP will be rejected with a redirect that clients must not follow automatically when the request contains credentials, since doing so could expose those credentials to intermediaries. The API accepts and returns JSON exclusively, with the exception of the bulk export endpoints, which stream newline-delimited JSON, and the media upload endpoints, which accept multipart form data in the manner described in the section on binary payloads later in this document.

Authentication

Every request to the Widget API must include an Authorization header bearing a valid API key or OAuth 2.0 access token, and the choice between these two mechanisms depends on whether you are building a server-side integration that acts on behalf of your own organization, in which case an API key is simpler and entirely sufficient, or a third-party application that acts on behalf of other users, in which case you must implement the full OAuth 2.0 authorization code flow with PKCE as described in RFC 7636. API keys are created in the dashboard under Settings, then API, then Keys, and each key is shown exactly once at creation time, so you must store it in a secret manager immediately because there is no mechanism for retrieving the plaintext value afterwards, only for rotating the key, which invalidates the previous value after a configurable grace period that defaults to twenty-four hours.

curl https://api.example.com/v2/widgets \
  -H "Authorization: Bearer wk_live_4eC39HqLyjWDarjtT1zdp7dc"

Tokens expire after one hour and must be refreshed using the refresh token grant, and note that refresh tokens are single-use in this API, meaning each refresh operation returns a new refresh token and invalidates the old one, a design known as refresh token rotation which limits the damage of a leaked token but which also means your token storage must be updated atomically after every refresh to avoid a situation in which two concurrent processes race to use the same refresh token and one of them is left holding an invalidated credential.

Rate limiting

The API enforces rate limits on a per-key basis using a token bucket algorithm with a sustained rate of one hundred requests per minute and a burst capacity of two hundred requests, and when you exceed these limits the API responds with HTTP status 429 and a Retry-After header indicating the number of seconds you should wait before retrying, which your client must honor because repeated violations within a rolling ten-minute window will result in a temporary suspension of the key lasting fifteen minutes. Every response includes the headers X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset so that well-behaved clients can pace themselves proactively rather than reactively, and we strongly recommend implementing exponential backoff with jitter for all retry logic rather than fixed-interval retries, since fixed intervals cause synchronized retry storms when many clients are affected by the same transient failure at the same time.

Pagination

All list endpoints are paginated using cursor-based pagination rather than offset-based pagination, a decision made because offset pagination produces inconsistent results when the underlying collection is modified between page requests, which happens frequently in practice. Each list response contains a data array and a next_cursor field, and when next_cursor is non-null you may pass it as the cursor query parameter of a subsequent request to retrieve the next page, continuing until next_cursor is null, which indicates that you have reached the end of the collection. The default page size is twenty-five items and the maximum is one hundred, controlled by the limit parameter, and note that the cursor encodes the sort order and filter parameters of the original request, so you must not change any other query parameters while paginating because the API will respond with error code cursor_mismatch if the parameters accompanying a cursor differ from those with which it was issued.

Error handling

Errors are returned with an appropriate HTTP status code and a JSON body containing a machine-readable code, a human-readable message, and where applicable a param field identifying the specific request parameter that caused the failure, and your integration should branch on the code field rather than parsing the message text, because messages are subject to change without notice and are not covered by our versioning policy. Client errors in the 4xx range indicate a problem with the request that will not be resolved by retrying, with the specific exceptions of 408, 409, and 429, which may be retried according to the guidance in the rate limiting section, while server errors in the 5xx range indicate a problem on our side that is usually transient, and all 5xx responses may be retried with exponential backoff provided the request is idempotent, which all GET, PUT, and DELETE requests are by definition, and which POST requests can be made to be by supplying an Idempotency-Key header containing a unique value that we will remember for twenty-four hours.

Endpoints

The core resource is the widget, accessed at /v2/widgets, which supports the standard verbs: GET on the collection lists widgets, POST creates a widget, GET on /v2/widgets/{id} retrieves a single widget, PATCH updates it partially, PUT replaces it entirely, and DELETE removes it, subject to the caveat that deletion is soft by default, moving the widget to a trashed state from which it can be restored for thirty days via the /v2/widgets/{id}/restore endpoint before being permanently purged by a background process whose timing is not guaranteed. Widget collections live at /v2/collections and behave analogously, with the additional nested route /v2/collections/{id}/widgets for managing membership, and note that a widget may belong to at most fifty collections simultaneously, a limit that exists for performance reasons and that will produce error code collection_limit_exceeded when violated.

POST /v2/widgets
{
  "name": "Flux capacitor",
  "sku": "FLX-88",
  "metadata": { "voltage": "1.21GW" }
}

Webhooks

Webhooks allow your application to receive notifications when resources change instead of polling, and you configure them in the dashboard or via the /v2/webhook_endpoints API by supplying a URL and a list of event types, after which we will deliver an HTTP POST to that URL for each matching event, signed with an HMAC-SHA256 signature carried in the Widget-Signature header, which you must verify using your endpoint's signing secret before trusting the payload, because anyone on the internet can send requests to your endpoint URL. Deliveries are retried with exponential backoff for up to seventy-two hours if your endpoint does not respond with a 2xx status within ten seconds, and events are delivered at least once but possibly more than once, so your handler must be idempotent, typically by recording processed event identifiers, and events may arrive out of order under retry conditions, so your handler must not assume that the sequence of deliveries matches the sequence of occurrences.

For questions not covered in this reference, consult the developer forum or open a support ticket, including in either case the request identifier from the X-Request-Id response header of any problematic call, since that identifier allows us to locate the corresponding trace in our logs within seconds rather than hours.
